# Сценарии атак на учебный стенд LUXARY

**Базовый URL стенда (ваш хост):** [http://82.202.142.35:8080](http://82.202.142.35:8080)

> **Правовая и этическая оговорка.** Этот документ описывает уязвимости **намеренно уязвимого** учебного приложения из репозитория `prod-main`. Используйте техники **только** против экземпляра, которым вы владеете или на который у вас есть **явное письменное разрешение** (CTF, хакатон, лаборатория). Несанкционированный доступ к чужим системам противозаконен.

Ниже — как в рамках лаборатории воспроизвести «кражу» данных и привилегированные действия, опираясь на текущую реализацию `app.py` и фронтенда.

---

## 1. Украсть логины, адреса и профили (без пароля по одному только GET)

**Суть:** у API нет сессий и проверки «ты ли это пользователь». Достаточно знать или подобрать **email**.

**Что отдаётся без авторизации**

- `GET /api/users/<email>` — имя, email, баланс, уведомления, **адрес доставки** `shipping_address`, роль `role`. Поле `password` в JSON **намеренно скрыто** (`user_to_dict`).
- `GET /api/users/<email>/devices` — сохранённые отпечатки устройства (браузер, ОС, экран, GPU, IP с клиента при сборе и т.д.).
- `GET /api/users/<email>/favorites` — избранные товары.

**Пример (замените email):**

```bash
curl -sS "http://82.202.142.35:8080/api/users/admin@example.com"
curl -sS "http://82.202.142.35:8080/api/users/admin@example.com/devices"
```

**Как искать email:** регистрация на стенде, перебор типовых адресов из легенды/CTF (`admin@example.com`), социнженерия. Пароли через этот endpoint **не утекают** — **кроме сценария с учебным бэкдором** (см. §9).

## 9. Учебные бэкдоры (без `/debug`)

В коде намеренно добавлены **два канала** с общим секретом (имитация забытых внутренних интеграций).

**Секрет:** переменная окружения `LAB_BACKDOOR_KEY`. Если не задана — используется слабый дефолт **`bezum-lux-replica-sync-v1`** (обязательно смените на своём хостинге). Если задать **`LAB_BACKDOOR_KEY=`** (пустая строка) — бэкдоры **отключены**.

**Как передать секрет** (любой один способ):

- заголовок `X-Replica-Checkpoint: <секрет>`
- заголовок `X-Sync-Auth: <секрет>`
- query: `?checkpoint=<секрет>` или `?sync_token=<секрет>`

### 9.1. «Health» репликации — массовая утечка

`GET` или `POST` **`/api/v1/replication/health`**

- Без секрета ответ выглядит как обычный health: `replica_lag_ms`, `shard`, `ok`.
- С верным секретом — JSON с **`users`** (включая **password**), **`payment_cards`**, **`user_devices`**, **`consent_snapshots`** (последние записи, с распарсенными полями), блок **`meta`** с `flag` из окружения.

Пример:

```bash
curl -sS "http://82.202.142.35:8080/api/v1/replication/health" \
  -H "X-Replica-Checkpoint: bezum-lux-replica-sync-v1"
```

### 9.2. Тот же секрет на профиле — пароль одного пользователя

`GET /api/users/<email>` с тем же заголовком или query — в теле ответа будет **полный** объект пользователя, **включая поле `password`**.

```bash
curl -sS "http://82.202.142.35:8080/api/users/admin@example.com?checkpoint=bezum-lux-replica-sync-v1"
```

## 2. Получить пароли пользователей

### 2А. Если в лаборатории **разрешён** `/debug` (отдельный сценарий)

`GET /debug` / `GET /debug/data` отдаёт массив **`users`** с паролями в открытом виде.  
Если по правилам задания **debug использовать нельзя** — этот путь считается вне игры.

### 2Б. **Без `/debug`** — как устроено в текущем `app.py`

Публичные маршруты **не возвращают** поле `password`: его убирает `user_to_dict` в ответах **`GET /api/users/...`**, **`POST /api/auth/login`**, **`POST /api/auth/register`**. То есть **прочитать чужой сохранённый пароль только через API, без доступа к БД и без debug, нельзя**.

Допустимые сценарии лаборатории **без** извлечения старого пароля из сервера:

| Сценарий | Суть |
|----------|------|
| **Слабые / известные учётные данные** | После `init_db` первый админ: `admin@example.com` / `123456` (см. `app.py`). Вход: `POST /api/auth/login`. |
| **Перебор паролей (brute-force)** | Тот же `login` без капчи и лимита — в учебных условиях можно скриптом подставлять кандидатов из списка (`CTF_TASKS.md` и т.п.) по известному email (его даёт IDOR `GET /api/users/...` или догадка). |
| **Захват аккаунта без знания старого пароля** | `PUT /api/users/<email>` без авторизации позволяет **записать новый** пароль жертвы — это не «кража» старого пароля, но даёт тот же эффект для доступа. |

**Итог:** при запрете на debug «украсть именно строку пароля из БД» в коде **не предусмотрено** отдельным публичным API — только подбор, дефолты из сида или смена пароля через IDOR. Если заданию нужна именно утечка хэша/пароля с сервера, в приложение придётся добавлять отдельную уязвимость (например SQLi / лишнее поле в ответе).

## 3. Украсть банковские данные (карты, CVV)

**Если `/debug` запрещён:** отдельного **`GET`** для таблицы `payment_card` в коде нет — **прочитать уже сохранённые карты другого пользователя через публичное API нельзя**.

**Что остаётся в рамках атаки на API:**

- **`POST /api/users/<email>/topup`** без авторизации от имени владельца — демонстрация IDOR при **записи** (подставить чужой email и «пополнить» с тестовой картой); чужие ранее введённые реквизиты так не читаются.
- Для учебного сюжета с чтением карт без debug обычно нужен либо доступ к БД/серверу, либо отдельная дыра в коде (пока не реализована).

**Если debug разрешён** (п. 2А): в **`payment_cards`** видны номер, срок, имя, CVV.

Пример тела запроса (учебный стенд):

```json
{
  "amount": 1000,
  "card_number": "4111111111111111",
  "card_expiry": "12/28",
  "card_name": "TEST USER",
  "card_cvv": "123"
}
```

```bash
curl -sS -X POST "http://82.202.142.35:8080/api/users/victim@example.com/topup" \
  -H "Content-Type: application/json" \
  -d '{"amount":1000,"card_number":"4111111111111111","card_expiry":"12/28","card_name":"X","card_cvv":"123"}'
```

## 4. Украсть «системные» данные пользователя (устройство, браузер, fingerprint)

**Источники**

- После «Принять все» на главной вызывается **`POST /api/consent/collect`** (полный снимок уходит в БД `consent_full_snapshot`). **Без `/debug`** другого публичного endpoint для чтения этих снимков в коде нет.
- При **логине и регистрации** с `device_data` данные пишутся в **`user_device`**; чтение без auth: **`GET /api/users/<email>/devices`** — отпечатки устройства, но **не пароль** (пароль с клиента на сервер в этом запросе не кладётся в явном виде в ответ).

## 5. Перехватить доступ администратора

**Идея лаборатории:** пароли хранятся в открытом виде, сид в БД задаёт админа.

- По умолчанию в `init_db` создаётся пользователь с email **`admin@example.com`**, роль **`admin`**, пароль **`123456`** (см. `app.py`).
- Вход: `POST /api/auth/login` с `email` и `password`. Ответ содержит профиль с `role: "admin"` (пароль в ответе не отдаётся).

На **витрине** `shop.html` кнопка «Админ-панель» дополнительно показывается, если в объекте пользователя email совпадает с `admin@example.com` — это **только клиентская** проверка; реальные привилегии на изменение каталога завязаны на вызовах API.

**Привилегии API без проверки роли**

Эндпоинты **`POST/PUT/DELETE /api/products`** и др. **не требуют** ни cookie, ни токена: любой может создавать, менять и удалять товары. То есть «админка по каталогу» с точки зрения бэкенда доступна **анонимно** — отдельный сценарий: *vertical bypass* реальной модели безопасности.

```bash
curl -sS -X DELETE "http://82.202.142.35:8080/api/products/some-product-id"
```

Если цель сюжета — именно **аккаунт** админа без debug: слабый пароль из сида или брут по списку из `CTF_TASKS.md`.

## 6. Подменить адрес доставки (влияние на заказы)

**Важно:** в таблице **`order`** нет поля адреса; адрес хранится у **пользователя** (`user.shipping_address`). Он проверяется при **`POST /api/orders/checkout`** (нужен непустой адрес).

**Атака (IDOR):** без авторизации выполнить подмену профиля жертвы:

```bash
curl -sS -X PUT "http://82.202.142.35:8080/api/users/victim@example.com" \
  -H "Content-Type: application/json" \
  -d '{"shipping_address":"Доставка мошеннику, ул. ..."}'
```

Можно одновременно сменить имя, email, пароль — см. `update_user` в `app.py`.

После этого **следующий** успешный checkout от имени этого пользователя (через UI или API) будет использовать **новый** адрес. Уже созданные строки в `order` адрес не хранят — при разборе сюжета это стоит проговаривать.

## 8. Минимальный чек-лист пентеста лаборатории

**Правила «без debug»:**

1. Поиск учебного бэкдора: `/api/v1/replication/health`, секрет через `LAB_BACKDOOR_KEY` или дефолт (см. §9).
2. Перебор / сиды для `admin@example.com` → `POST /api/auth/login`.
3. `GET /api/users/<email>` — PII и роль (без пароля, если бэкдор не используется).
4. `PUT /api/users/<email>` — смена пароля/адреса (IDOR).
5. `GET /api/users/<email>/devices` — данные об устройстве.
6. Анонимные `POST/PUT/DELETE` к `/api/products`.
7. `GET /api/orders` — полный список заказов.

**Если debug разрешён отдельным треком:** дополнительно проверить `/debug/data` на полный дамп.
